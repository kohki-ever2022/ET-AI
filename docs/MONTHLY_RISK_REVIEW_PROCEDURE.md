# 月次リスクレビュー実施手順

**目的**: ET-AIプロジェクトのリスクを定期的に見直し、適切な対策を講じる
**頻度**: 月次（毎月第1金曜日）
**所要時間**: 2時間
**責任者**: プロジェクトマネージャー

## 目次

1. [事前準備](#事前準備)
2. [レビューの進め方](#レビューの進め方)
3. [レビューテンプレート](#レビューテンプレート)
4. [チェックリスト](#チェックリスト)
5. [フォローアップ](#フォローアップ)

---

## 事前準備

### 1週間前

**プロジェクトマネージャー**:
- [ ] 会議の日程確認（毎月第1金曜日 14:00-16:00）
- [ ] 参加者への招待送信
- [ ] 前回のアクションアイテムの進捗確認

### 3日前

**各チームリーダー**:
- [ ] 担当リスクのステータス更新
- [ ] モニタリング指標の収集
- [ ] 新規リスクの洗い出し

### 前日

**プロジェクトマネージャー**:
- [ ] レビュー資料の準備
  - 現在のリスクリスト
  - 前月からの変更点
  - モニタリング指標データ
  - インシデントレポート
- [ ] 会議室/オンライン会議の確認
- [ ] 資料の事前共有（Slack/メール）

## レビューの進め方

### タイムテーブル（120分）

```
14:00-14:15 (15分) - オープニング
14:15-14:30 (15分) - アクションアイテム確認
14:30-15:00 (30分) - 既存リスクのステータス更新
15:00-15:20 (20分) - 新規リスクの識別
15:20-15:35 (15分) - インシデントレビュー
15:35-15:55 (20分) - モニタリング指標レビュー
15:55-16:00 (5分)  - クロージング
```

### セッション詳細

#### セッション1: オープニング（15分）

**目的**: レビューの目的と流れを共有

**進行**:
1. **アジェンダの確認** (2分)
2. **前回からの主要な変更点** (5分)
   - プロジェクトの進捗状況
   - 環境の変化（技術、組織、市場）
3. **今回の重点テーマ** (5分)
   - 特に注目すべきリスク領域
   - 新機能リリース予定
4. **質疑応答** (3分)

**成果物**: なし

---

#### セッション2: アクションアイテム確認（15分）

**目的**: 前回のアクションアイテムの完了状況を確認

**進行**:
1. **完了したアクション** (5分)
   - 実施内容の簡単な報告
   - 効果の確認

2. **進行中のアクション** (5分)
   - 進捗率の確認
   - 障害の有無
   - デッドラインの再確認

3. **未着手のアクション** (5分)
   - 未着手の理由
   - 優先度の再評価
   - デッドラインの再設定

**使用ツール**: アクションアイテムトラッカー

**成果物**: 更新されたアクションアイテムリスト

---

#### セッション3: 既存リスクのステータス更新（30分）

**目的**: 全リスクの現状を確認し、評価を更新

**進行**:

各カテゴリーごとにレビュー（各6分）：
1. 技術リスク
2. スコープリスク
3. 人的リスク
4. 外部依存リスク
5. 運用リスク

**各リスクの確認項目**:
```
□ 発生確率は変わったか？（高/中/低）
□ 影響度は変わったか？（高/中/低）
□ 予防策は実施されているか？
□ 予防策は効果的か？
□ モニタリング指標は適切か？
□ 責任者は変わったか？
□ リスクをクローズできるか？
```

**更新が必要な項目**:
- リスクスコアの再計算
- 対策の有効性評価
- 優先度の変更
- 責任者の更新

**議論のポイント**:
- スコアが上昇したリスク → なぜ？追加対策は？
- スコアが低下したリスク → 対策の効果検証
- 長期間変化がないリスク → 監視方法は適切か？

**成果物**: 更新されたリスクリスト

---

#### セッション4: 新規リスクの識別（20分）

**目的**: 新たに発生した、または認識されたリスクを特定

**進行**:

**ステップ1: ブレインストーミング** (10分)

以下の観点から新規リスクを議論：

1. **技術環境の変化**
   - 新しい依存ライブラリやツールの導入
   - インフラ構成の変更
   - 新機能の実装
   - パフォーマンスの変化

2. **外部環境の変化**
   - 外部サービスのアナウンスメント
   - 価格改定の予告
   - 法規制の動向
   - セキュリティ脆弱性の公開

3. **プロジェクトの変化**
   - チーム構成の変更
   - スコープの追加/変更
   - 予算の変更
   - スケジュールの変更

4. **ユーザーフィードバック**
   - 新たな要望
   - 不満や問題の報告
   - 使用パターンの変化

**ステップ2: リスク評価** (5分)

各新規リスクについて：
- リスク内容の明確化
- 発生確率の評価（高/中/低）
- 影響度の評価（高/中/低）
- リスクスコアの計算

**ステップ3: 対応方針の決定** (5分)

- 即座に対応が必要か？
- 次月のアクションアイテムに追加すべきか？
- 継続監視で十分か？
- 責任者の割り当て

**成果物**: 新規リスクリスト（評価済み）

---

#### セッション5: インシデント・問題のレビュー（15分）

**目的**: 前月発生したインシデントを振り返り、学びを得る

**進行**:

各インシデントについて以下を確認：

**基本情報** (2分/インシデント):
- インシデント概要
- 発生日時
- 重大度（P0/P1/P2/P3）
- 影響範囲

**分析** (3分/インシデント):
- 根本原因は何だったか？
- 対応は適切だったか？
- 対応時間は目標内だったか？
- 予防できたか？

**学び** (2分/インシデント):
- 新たなリスクとして追加すべきか？
- 既存リスクの評価を変更すべきか？
- 予防策や対応策を強化すべきか？
- ドキュメントを更新すべきか？

**成果物**: インシデントレビューサマリー

---

## よくある質問（FAQ）

### Q1: レビューに出席できない場合は？

**A**:
1. 事前に担当リスクのステータスをSlackまたはメールで共有
2. 代理出席者を指名
3. レビュー後のレポートを確認し、アクションアイテムを確認

### Q2: 緊急リスクが発生した場合は？

**A**:
月次レビューを待たずに、即座に以下を実施：
1. Slack #risksチャンネルで報告
2. 緊急ミーティングの招集（必要に応じて）
3. 暫定対策の実施
4. 次回レビューで正式に追加

### Q3: リスクをクローズする基準は？

**A**:
以下のいずれかを満たす場合：
1. 予防策が完了し、発生確率がゼロになった
2. プロジェクトのスコープ変更により該当しなくなった
3. 外部環境の変化により該当しなくなった

注意: 安易にクローズせず、慎重に判断する

### Q4: モニタリング指標が目標を超えた場合は？

**A**:
1. 即座にアラート通知（自動）
2. 原因分析
3. 短期的な対応（閾値内に戻す）
4. 次回レビューで中長期的な対策を議論

### Q5: アクションアイテムが期限内に完了しない場合は？

**A**:
1. 早めに（デッドラインの1週間前）報告
2. 理由を明確化
3. 新しいデッドラインの設定
4. 優先度の再評価
5. 必要に応じてリソースの再配分
